◆ EMV_CT_ApplyConfiguration()

DLL_CTC EMV_ADK_INFO EMV_CT_ApplyConfiguration ( unsigned long  options)

Write configuration to files.

This function writes the configuration to file system if EMV_CT_INIT_OPT_CONFIG_MODE was set. If EMV_CT_INIT_OPT_CONFIG_MODE was not set, this function returns EMV_ADK_NO_EXEC and does nothing.

Explanation in programmers guide: Virtual terminal mapping table

Author
GSS R&D Germany
Parameters
[in]optionsRFU
TLV tag TAG_DF8F0F_APPLYCONFIG_OPTIONS
Returns
EMV_ADK_OK: success
EMV_ADK_INTERNAL: internal communication problem, illegal file content, memory allocation
EMV_ADK_TLV_BUILD_ERR: TLV processing problem
EMV_ADK_NOT_ALLOWED: concurrent call or not allowed within callback
EMV_ADK_SAVE_ERROR: TermData/AppliData/CapKeys/Hotlist/VirtTermMap file persistence problem
EMV_ADK_NO_EXEC: maximum number of configurable AIDs exceeded
Serialization:
Class: CLA_EMV, Instruction: INS_APPLY_CFG

◆ EMV_CT_GetCAPKeyInfo()

DLL_CTC EMV_ADK_INFO EMV_CT_GetCAPKeyInfo ( EMV_CT_CAPREAD_TYPE *  pxKeyInfo,
unsigned  capacity,
unsigned  offset,
unsigned *  received,
unsigned *  configured 
)

Reads CAP key information.

Author
GSS R&D Germany

Explanation in programmers guide: Configure CAP Keys

Parameters
[out]pxKeyInfoCAP key info data (buffer length at least 7 times capacity)
see EMV_CT_CAPREAD_STRUCT
TLV tag TAG_ISO_DATA
[in]capacitynumber of keys that can be stored in pxKeyInfo
TLV tag TAG_KEY_NUMBER
[in]offsetfor reading the key list in several chunks
TLV tag TAG_DF3B_PARAMETER_1
[out]receivednumber of data sets stored to pxKeyInfo
[out]configurednumber of configured keys
TLV tag TAG_KEY_NUMBER
Returns
EMV_ADK_OK: success
EMV_ADK_PARAM: bad pxKeyData or pucMaxnum input params
EMV_ADK_INTERNAL: internal communication problem, illegal file content, memory allocation
EMV_ADK_TLV_BUILD_ERR: TLV processing problem
EMV_ADK_NOT_ALLOWED: concurrent call or not allowed within callback
EMV_ADK_READ_ERROR: CapKeys file corruption
Note
Currently maximum number of CAP keys to be stored is 512.

Serialization: As this command uses the same class and instruction as EMV_CT_ReadCAPKeys() the presence of parameter offset marks the support of compressed response format. Legacy frameworks will ignore this parameter and return key info in TLV format.
Request
Class: CLA_EMV, Instruction: INS_CAPKEY_CFG, P2: P2_GET
capacity TAG_KEY_NUMBER, up to 255, for more a loop is required
offset TAG_DF3B_PARAMETER_1, length 2
Response
configured TAG_KEY_NUMBER, length 2 indicates compressed format, legacy framework will send length 1
key info TAG_ISO_DATA sequence of RID, index and key length up to given capacity, data length is 7 times number of keys info blocks while the number of blocks is min(capacity, number of stored keys - offset)

Get configured application data.

This function can only be used if init option EMV_CTLS_INIT_OPT_NEW_CFG_INTF was set.

Warning
struct members of type EMV_CTLS_DATA_TYPE are allocated within libEMV_CTLS_Client. The next call to this function as well as a call to Exit Framework will invalidate the data pointers of the first result. For the same reason, this function is not thread-safe

Transport tags used in serialization: CLA_EMV INS_APPLI_CFG P2_GET

GSS R&D Germany

Explanation in programmers guide: Configure CTLS Terminal data

Author
GSS R&D Germany
Parameters
[in]pxTermData... terminal data, see EMV_CTLS_TERMDATA_STRUCT
Returns
EMV_ADK_OK: success
EMV_ADK_PARAM: bad pxTermData input param
EMV_ADK_TLV_BUILD_ERR: TLV processing problem
EMV_ADK_INTERNAL: Cless Cfg data not init or memory allocation problem
EMV_ADK_NOT_ALLOWED: concurrent call or not allowed within callback
EMV_ADK_READ_ERROR: TermData file corruption
EMV_ADK_SAVE_ERROR: TermData file persistence problem
Serialization:
Class: CLA_EMV or CLA_EMV_ALT, Instruction: INS_TERM_CFG, P2: P2_SET
